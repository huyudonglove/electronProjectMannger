# 项目协作数据结构

Electron Manager 使用 Markdown 作为项目协作的主数据源。JSON 只保存配置和可重新生成的索引，不承担人工维护的业务事实。

## 当前结构

```text
<manager-data-root>/projects/<project-id>/
  project.json
  agent-brief.json
  index.json
  record-counters.json
  versions/
    版本索引.md
    V001/
      工程任务.md
      想法与问题.md
      研究.md
      待确认事项.md
      风险与后续.md
      工作记录/
        YYYY-MM.md
  collaboration/
    当前项目基线.md
    数据层规范.md
    Agent 同步交接.md
    需求变更索引.md
  constraints/
    项目约束.md
  documents/
  skills/
    project-collaboration/
      SKILL.md
```

## 版本边界

- `versions/Vxxx/` 是任务、想法、研究、协作问题、风险和工作记录的物理边界。
- 当前版本由 `project.json.currentVersionId` 指定。
- `agent-brief.json.currentDataPaths` 给出当前版本的准确写入路径。
- 已完成版本默认只读；只有追溯历史或当前上下文不足时才读取。
- 文档和项目约束属于项目整体，`version:: Vxxx` 只表示来源，不参与版本筛选。

## 协作记录

- 任务使用 `Txxx`，是执行和验收的基本单位。
- 想法使用 `Ixxx`，处理后必须写明回答、关联任务或不处理原因。
- 研究使用 `Dxxx`，以 `mode:: breadth | depth` 区分广度和深度，以 `status:: pending | doing | done | archived` 表示生命周期。
- 协作线程使用稳定 `Qxxx`，保存用户与 Agent 的连续对话和下一步责任。
- 风险使用 `Rxxx`，保存验证限制、技术风险和后续工作，不进入双向协作线程。
- 工作记录使用 `Lxxx`，按版本和月份保存；只要修改源码、配置、测试、项目文档、知识条目或协作规则等交付文件，就必须记录。
- 工作记录使用 `record_level:: light | standard | deep`：轻量改动只写目标、产出、修改文件、验证、验收结果和风险；正常开发使用 standard；研究、架构和长期规则变更使用 deep。

问题状态固定为：

- `open`：待用户回复。
- `decided`：待 Agent 回复或落实。
- `resolved`：线程已完成。
- `expired`：仅由 Agent 或系统归档已被替代、确实无关的线程，不作为用户日常操作。

每条用户或 Agent 回复都追加到问题的 `### 对话记录`，不覆盖历史消息。用户新建线程或回复后，状态进入 `decided`；Agent 追问时进入 `open`；确认协作完成后进入 `resolved`。已完成或归档的线程仍可继续讨论，新的用户回复会重新进入 `decided`。

风险状态固定为：

- `open`：仍需关注。
- `resolved`：已处理。
- `expired`：已无关。

## 研究模式

- `breadth`：广度研究。至少覆盖三条实质不同的路径、方案或视角，横向比较依据、优缺点、适用条件和未知项，形成候选排序，并指出值得转入深度研究的方向。
- `depth`：深度研究。聚焦一个明确对象或路径，追踪证据、源码或实现细节、约束与反例，验证关键结论，并记录置信度、剩余缺口、风险和下一步。
- 广度研究在继续搜索不再产生新的重要类别时停止；深度研究在核心问题已有证据支撑，或阻塞条件已明确记录时停止。
- 新建研究只创建 pending D，不提前生成 W 或 L；pending 和 doing 进入 `agent-brief.json.activeResearch`。
- Agent 领取后改为 doing；短结果直接写回 D，较长结果完成后才创建关联 W；完成后改为 done，并只写一条 L 工作记录。
- 运行时不自动迁移旧字段；旧项目需要迁移时，按[旧项目迁移说明](./旧项目迁移说明.md)单独处理。
- 明确的修改要求、缺陷和执行待办进入任务，不进入研究。

## 写入规则

- 所有项目记录必须包含 `version:: Vxxx`。
- 记录按短 ID 倒序保存，较大的 ID 写在较小 ID 前面。
- 写入通过项目级队列串行化，并使用临时文件原子替换。
- `record-counters.json` 持久保存各类记录的最大编号，删除记录后不复用 ID。
- Markdown 变化后重新生成 `agent-brief.json`、`当前项目基线.md` 和 `index.json`。
- 新记录只写入 `agent-brief.json.currentDataPaths` 指向的当前版本文件。

## 工作记录语义

- standard/deep 工作记录保留完整执行和验收信息，并标注 `record_level:: standard | deep`；light 工作记录标注 `record_level:: light`，只保留 `### 用户目标`、`### 产出`、`### 修改文件`、`### 验证`、`### 验收结果`、`### 已知风险`。
- 任务状态、问题回复、研究状态和自动生成的 brief/index/基线属于协作元数据或派生缓存，不单独触发工作记录，也不递归生成日志。
- `### 验收标准` 位于产出和执行之前，是 Agent 开始工作前必须读取的完成口径（standard/deep 必需，light 可在简单改动中省略）。
- `### 验证` 记录实际执行的检查、命令和结果，不代替验收判断。
- `### 验收结果` 位于验证之后，逐项说明验收标准为通过、部分通过或未通过。
- `### 已知风险` 和 `### 后续事项` 记录不影响本次验收结论、但仍需保留的事项。
- 历史工作记录按标题解析，不要求为此次顺序调整批量改写。

## Agent 启动顺序

1. 读取项目根目录的 `.agent-collaboration.md`，定位管理数据目录。
2. 读取 `agent-brief.json` 和 `collaboration/当前项目基线.md`。
3. 读取 `skills/project-collaboration/SKILL.md`。
4. 默认只读取 `currentDataPaths` 指向的当前版本记录。
5. 需要长期规则时读取 `constraints/项目约束.md`。
6. 只有当前上下文不足或用户明确要求时才进入历史版本。

应用启动时会统一刷新所有已登记项目的基线、数据规范、交接说明、SKILL 和派生 JSON；不会借此迁移或补写旧项目的业务 Markdown。
