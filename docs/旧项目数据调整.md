# 旧项目数据调整方式

Electron Manager 当前使用 schema 3。任务、想法、研究、协作问题、风险和工作记录按版本聚合存储；项目文档和项目约束仍是项目级资料，只用版本号追溯来源。

## 自动迁移

打开 schema 1 或 schema 2 项目时，应用会执行一次迁移：

1. 读取记录上的 `version:: Vxxx`；缺失时归入 `project.json.currentVersionId`。
2. 将任务、想法、研究、问题和风险写入对应的 `versions/Vxxx/` 文件。
3. 将工作记录按版本和 `created:: YYYY-MM` 拆到月度文件。
4. 对每类记录进行数量校验。
5. 校验通过后删除旧的 `tasks/`、`thoughts/`、`research/`、`work-logs/` 大文件和旧协作问题文件。
6. 将 `project.json.schemaVersion` 更新为 `3`，重新生成基线、索引、brief 和 SKILL。

迁移失败时不会更新 schema 版本，也不会进入删除旧文件的步骤。对重要项目仍建议在升级前保留整个数据目录备份。

## 记录要求

- 每条项目记录必须有稳定短 ID 和 `version:: Vxxx`。
- `Qxxx` 只记录需要用户决定、澄清或解除阻塞的问题，状态为 `open`、`decided`、`resolved` 或 `expired`。
- `Rxxx` 记录验证限制、技术风险和后续事项，状态为 `open`、`resolved` 或 `expired`。
- 任务、想法和工作记录不再使用内嵌“未确认事项”作为协作问题来源。
- 文档 `Wxxx` 和项目约束 `Cxxx` 带来源版本，但不会随版本筛选隐藏。
- 工作记录必须包含用户目标、需求理解、产出、关键步骤、验证、验收标准、已知风险和后续事项。
- 已完成版本默认作为只读历史；如果旧问题重新影响当前阶段，应在当前版本建立新记录并引用旧 ID。

## 完成检查

- `project.json.schemaVersion` 为 `3`。
- `agent-brief.json.currentDataPaths` 指向当前 `versions/Vxxx/` 目录。
- 每个版本都有任务、想法、研究、问题、风险文件。
- 工作记录位于版本内的月度文件。
- 旧项目级任务、想法、研究和工作记录文件已删除。
- `agent-brief.json`、`index.json` 和 `collaboration/当前项目基线.md` 已重新生成。
- 记录总数与迁移前一致。
